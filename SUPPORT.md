# Support
I am a solo developer, and my time is best spent optimizing code and building features. Before you ask for help, you must follow this path to ensure you aren't wasting a cycle.

## 1. Self-Help (Check the Docs First)
Most "problems" are just people not reading the manual. Before opening an issue, read these:
*   **README.md:** For installation and basic usage.
*   **ARCHITECTURE.md:** For technical details on how the code works.
*   **The Wiki:** For deep dives into specific functions.
*   **Existing Issues:** Use the search bar in the "Issues" tab. If someone else already asked it, I won't answer it again.

## 2. Where to get help
The **only** official place for support is the **GitHub Issues** tab. 
*   **Do not** send me private messages.
*   **Do not** email me for basic usage questions.
*   Keep all technical talk in the public repository so others can see the solution.

## 3. How to ask
If you have checked the docs and searched the issues, and you still have a problem, open a new issue using the appropriate template. 

**Your request must be:**
*   **Specific:** "It doesn't work" is not an issue. "Function X returns Y when I expected Z" is an issue.
*   **Labeled:** Use the labels I defined in the `CONTRIBUTING.md` (e.g., `BUG:` or `FEEDBACK:`).
*   **Clear:** If I can't understand you, I will close the issue.

## 4. Response Time
I work on this project when I have time and interest. I do not have a "support team." I will respond when I see a constructive, well-documented request that adds value to the project.

---
**Efficiency is the goal. Help yourself first.**
